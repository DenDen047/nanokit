# アルゴリズム概要資料の作成ガイド

CTO やステークホルダーが短時間で全体像を把握できる、アルゴリズム・設計思想の概要資料を作成する。

## 対象読者

- CTO / テックリード / 投資家 / 事業責任者
- コードは読まないが、ロジックの妥当性や設計方針を判断する人
- 10-15 分で読み切れることを前提とする

## 資料の構成

```
# プロジェクト名 アルゴリズム概要

> 一文の位置付け説明（誰向け、何の資料か）

## 一言で言うと             ← 1-2 文でシステム全体を要約
## 全体の流れ               ← パイプライン / フロー図
## 各ステップの詳細          ← ステップごとに 設計の意図 付き
## 段階的な拡張計画（任意）   ← ロードマップ
## セキュリティ・リスク（任意）← 安全機構
```

## 各セクションの書き方

### 1. 一言で言うと

システムの目的を **技術用語なしで** 1-2 文にまとめる。
「何を入力して、何が出力され、ユーザーにどんな価値があるか」を伝える。

```markdown
ユーザーが Komga (漫画) と Immich (画像) で示した好みを自動学習し、
外部サイトの新着から好みに合うコンテンツを選別して自動インポートする。
```

### 2. 全体の流れ

パイプライン全体を **一行の流れ** と **図** で示す。
ここで読者は「何ステップあるか」「各ステップの役割」を瞬時に把握する。

**一行の流れ** (番号付きで左から右):
```
1. 嗜好を学ぶ → 2. 削除を検知 → 3. 新着を採点 → 4. 厳選する → 5. 取得・配置
```

**ASCII フロー図**: Mermaid や画像ではなく、テキストベースの図を使う。
どの環境でも表示崩れしない。入力ソース → 処理ブロック → 出力先の関係を示す。

### 3. 各ステップの詳細

ステップごとにセクションを分け、以下の構造で書く:

```markdown
## N. ステップ名 (内部コンポーネント名)

1行の概要説明。

### コアメトリクス / ロジック

数式や疑似コードで、判断基準を示す。
コードそのものは書かない。計算の意味がわかる粒度にする。

> 設計の意図: なぜこの方式を選んだか、何を優先したかを 2-3 文で。
```

#### 「設計の意図」の書き方

各ステップの末尾に `> 設計の意図:` ブロックを置く。これが資料の核。

良い設計の意図の例:
- 「ユーザーに追加の操作負担をかけない」(ユーザー体験)
- 「ロジックの透明性と説明可能性を確保」(保守性)
- 「量より質。レビュー負荷を最小化」(運用)
- 「削除という自然な行動を学習に利用」(データ活用)

悪い例:
- 「SQLite を使用しているため」(実装詳細)
- 「パフォーマンスのため」(曖昧)

### 4. 表の活用

パラメータ、ルール、比較は **テーブル** で示す。文章より圧倒的に早く読める。

```markdown
| ルール | デフォルト | 目的 |
|--------|-----------|------|
| スコア閾値 | 0.1 以上 | 確度の低い推薦を排除 |
| 日次上限 | 50 冊/日 | ストレージとレビュー負荷を制限 |
```

**具体例の表** も効果的。抽象的な数式よりも、具体的な数値例が理解を助ける:

```markdown
| 作者 | RL 作品 | 全作品 | hit_rate | 最終スコア |
|------|---------|--------|----------|-----------|
| A    | 4       | 4      | 1.0      | 1.5       |
| B    | 1       | 4      | 0.25     | 0.25      |
```

### 5. 段階的な拡張計画 (任意)

Phase 形式でロードマップを示す。現在のフェーズを強調する。

```markdown
| Phase | 状態 | 手法 |
|-------|------|------|
| **1** | **稼働中** | ルールベース |
| 2 | 未着手 | タグ類似度 |
| 3 | 未着手 | Embedding |
```

### 6. セキュリティ・リスク (任意)

安全機構を箇条書きで。詳細な仕組みではなく「何を守っているか」を伝える。

## 原則

### 書くこと

- **Why (なぜ)**: 各設計判断の理由
- **What (何を)**: 各ステップが何をするか
- **数式・具体例**: 判断基準を数値で示す
- **トレードオフ**: 何を優先し、何を犠牲にしたか

### 書かないこと

- **How (どうやって)**: コードの実装方法、クラス名、関数名
- **API エンドポイント**: REST API の仕様
- **設定ファイル**: `.env` の変数名、config のキー名
- **インフラ構成**: サーバー IP、LXC ID、デプロイ手順
- **ライブラリ名**: httpx, pydantic などの技術スタック

### 文体

- **日本語**: 社内資料として自然な文体
- **断定調**: 「〜している」「〜する」(「〜と思われる」は避ける)
- **短文**: 1 文 40 文字以内を目安に
- **見出しは動詞**: 「嗜好プロファイル構築」ではなく「嗜好を学ぶ」

## 作成手順

1. **コードを読む**: 主要なパイプライン / エントリポイントからたどる
2. **ステップを列挙**: パイプラインの各段階を特定する
3. **コアメトリクスを抽出**: 各ステップの判断基準 (数式、閾値) を特定する
4. **設計意図を言語化**: 「なぜこの方式か」を各ステップごとに書く
5. **具体例を作る**: 数値例のテーブルを作成する
6. **実装詳細を削る**: コード由来の情報 (クラス名、変数名) を除去する

## アンチパターン

- **設計書の要約**: design.md をそのまま縮めただけ。実装詳細が残っている
- **コード付き**: Python/SQL のスニペットが含まれている
- **判断基準なし**: 「スコアリングする」だけで、何をどう計算するか不明
- **設計意図なし**: 「何をするか」だけで「なぜそうするか」がない
- **長すぎる**: 20 ページ超。10-15 分で読めない
