TL;DR

あなたの環境はすでにかなり作り込まれた多層構成で、今回はゼロから作るのではなく 既存資産の再編・拡張です。鍵になる気づきは1つ — 「アカウント」は2つの独立した軸でできていること。 ① Claude のログイン/課金アカウント（CLAUDE_CONFIG_DIR で切替）と、 ② データアカウント（Google/Slack/Notion のトークンで切替）。 この2軸が独立だと分かると、「HDTからはHDTのClaudeアカウント、でも他フォルダからHDTの予定も触りたい」が 矛盾なく成立することが見えてきます。

このあと何をするか: 3つの狙いを、リポジトリの責務に沿って配置します。 狙い②=HDT用 Google MCP を常設して Claude が用途でデータアカウントを振り分け／ 狙い①=視覚確認フックを拡張＋強命令＋専用コマンド／ 狙い③=終了フックで好みを抽出し全プロジェクト共通のメモリへ。 まず §2 の原理を読めば、残りは「その原理をどう広げるか」の話になります。
本書の用語（「アカウント」は必ず修飾します）:
  • Claudeアカウント … Claude Code のログイン / 課金（軸A）。個人 Max / HDT Team。切替は CLAUDE_CONFIG_DIR。
  • データアカウント … Google / Slack / Notion などサービス側のアカウント（軸B）。具体的には Googleアカウント / Slackアカウント / Notionアカウント と書きます。
以降、「アカウント」とだけは書かず、必ず上のどちらか（または具体的なサービス名）を明記します。

1. 全体像 — 3リポジトリの役割 誰が何を持っているか

まず登場人物の整理です。あなたの Claude Code 設定は3つのリポジトリに役割分担されています。 ここを取り違えると「どこを直せばいいか」が分からなくなるので、最初に固定します。

リポジトリ役割反映方法主な中身
nanokit/claude/グローバル唯一の真実（全プロジェクト共通）dotter で ~/.claude/ へ symlinksettings.json・hooks・skills・agents・CLAUDE.md・共有HTTP MCP
claude-settings/org（会社/案件）別テンプレと横断実行各 org dir に配置（dotter外）org別 .envrc・.mcp.json・cross-org-search スキル
実行ディレクトリ群
(HDT/, frogiraffe/…)		データ実体とスコープdirenv が .envrc を自動適用そのスコープのデータアカウント用トークン・creds と、Claudeアカウント切替の CLAUDE_CONFIG_DIR
原則: 設定の変更は必ず nanokit 側を編集 → dotter deploy。 ~/.claude/ を直接いじると nanokit のワーキングツリーが変わります（symlink のため）。

2.【基礎】複数のデータアカウントの使い分けの原理 ここが一番大事

「複数の Google / Slack / Notion アカウントをどう使い分けているのか分からない」という所を、 仕組みから説明します。結論を先に言うと —

原理ひとことで: あなたが今どのディレクトリにいるかで、使うデータアカウントが自動的に決まります。 仕掛けは direnv。ディレクトリに入った瞬間、そこに置かれた .envrc が 「どのトークン・どの認証情報を使うか」を環境変数にセットし、MCP サーバはその環境を見て起動するからです。

ステップで追う（①→⑥）

  1. ① OS Keychain — 全データアカウントの秘密（APIキー/トークン）は OS のキーチェーンに1か所で保管。コードには絶対に書きません。
  2. ② cd したディレクトリ — あなたが ~/nanokit にいるか HDT/ にいるかが「スコープ」を決める入口です。
  3. ③ direnv — ディレクトリに入った瞬間、そこの .envrc を自動実行。Keychain から必要な秘密だけ取り出して環境変数にセットします。
  4. ④ 環境が分岐 — 個人スコープなら個人の creds／HDTスコープなら HDT の creds。HDTでは NOTION_TOKEN を 明示的に unset して個人トークンの漏洩を防ぎ、CLAUDE_CONFIG_DIR=~/.claude-hdt をセットします。
  5. ⑤ MCPサーバ — Google/Slack/Notion などの MCP サーバは、その時の環境変数を見て起動。だから「個人の env で起動したサーバ＝個人のデータアカウントのツール」になります。
  6. ⑥ Claude が見えるツール — 結果として Claude Code が触れるツール集合が「今いるスコープのデータアカウント」に一致します。Claude は魔法でデータアカウントを選んでいるのではなく、環境が先に決まっているのです。

2つの独立した「アカウント」軸 — ここが混乱の元

「アカウント」という1語に、実は性質の違う2つが混ざっています。これを分けて考えると一気にクリアになります。

軸何のアカウントか切替スイッチ例
軸AClaude のログイン/課金（どの Claude プランで動くか）CLAUDE_CONFIG_DIR個人 Max（~/.claude）／HDT Team（~/.claude-hdt）
軸Bデータ（どの Google/Slack/Notion を読み書きするか）トークン / creds ディレクトリ個人 Google／HDT Google／frogiraffe Slack
核心の腑落ちポイント: 軸Aと軸Bは独立。だから「個人の Claude アカウントで動かしながら、HDT の Google カレンダーに予定を書く」は何の矛盾もなく成立します。 あなたの要望「HDTからはHDTのClaudeアカウントを使う」は軸Aの話、「他フォルダからもHDTの予定を触りたい」は軸Bの話で、別レイヤーなのです。
現状の制約: いま workspace-mcp（Google用）は個人1インスタンス（:8322）のみ。 つまり個人ディレクトリから HDT の Google（軸B）に手が届きません。これが狙い②で埋めるギャップです。

3. 狙い② マルチアカウント改良 Claudeが用途で振り分け + HDTもどこからでも

要望は「どのプロジェクトからでも複数のデータアカウントを確認したい。アクセス先は Claude が判断。HDTの予定調整も他フォルダからやりたい」。 §2の原理を踏まえると、やることは2つだけです。

(1) HDT用 Google MCP を常設する 最優先・予定調整を直接解決

既存の個人版（workspace-mcp-personal-server.sh, :8322）の姉妹として、 HDT の Google 認証を指す workspace-mcp-hdt-server.sh（:8324）を作り、user スコープに別サーバ名で登録します。

個人: mcp__workspace__* (:8322, creds=google-workspace/personal) HDT : mcp__workspace-hdt__* (:8324, creds=google-workspace/HDT) ← 新規

ツール名の頭が workspace と workspace-hdt で衝突しないため、 Claude は「個人カレンダー」と「HDTカレンダー」を別物として認識し、 両方に予定を入れられます。しかもこれは Google OAuth が書くだけなので、 軸A（Claudeアカウント＝ログイン/課金）の分離を一切壊しません。

(2) Claude に「データアカウント地図」を渡す “Claudeが選ぶ”の実体

nanokit/claude/CLAUDE.md に下表のようなデータアカウント地図を書きます。 Claude はこれを読んで「この依頼は個人＋HDT両方のカレンダーが要る」「これは frogiraffe の Slack」と自分でルーティングできます。

アカウント種別到達手段主用途
個人(=frogiraffe) Google常設mcp__workspace__* (:8322)個人/frogiraffe の予定/メール/Drive
HDT Google常設（新）mcp__workspace-hdt__* (:8324)HDTの予定/メール（他フォルダからも可）
frogiraffe Slack/Notion常設（直接・新）専用サーバ名 mcp__slack-frogiraffe__* 等・どのフォルダからもfrogiraffe 案件
HDT Slack/NotionHDT専用Team Connectors（~/.claude-hdt）cross-org-search 子経由で読取
全org横断オンデマンドcross-org-search スキル「全社の今週の予定/未読」等
設計原則（ご指摘を反映）: タイプA（ポータブルなトークン/OAuth）のデータアカウントは、すべて専用MCPサーバ名で「直接」アクセスできるようにする（cross-org-search を経由しない）。 cross-org-search は (a) タイプB の HDT Slack/Notion（Connector束縛で直接化できない） と (b) 多orgの一括集約（「全社の今週」等）だけに限定します。
※トレードオフ: 直接化したデータアカウントは全セッションでツールが見える（遅延ロードなので軽い）。よく横断参照するものだけ直接化し、稀にしか使わない org は cross-org-search 側に残すのが実用的です（詳細は直下の「常駐ポリシー」）。

org が8つに増えても破綻しない「常駐ポリシー」 スケール設計

あなたの org は 7つ（HDT / frogiraffe（=個人）/ EmplifAI / Lagoon / OpenHeart / PhD / uSonar）。frogiraffe を個人用としても使っているため personal は別建てしません。 frogiraffe は一例で、OpenHeart・uSonar 等も同様に必要です。ただし全orgを常駐（直接）にすると十数個のMCPサーバ＋全セッションでツールが激増して破綻します。そこで到達手段を3層に整理します。

層対象到達手段常時コスト
Tier1 常駐・直接横断頻度が高い少数（例: frogiraffe=個人/HDT Google）専用サーバ常駐（HTTPシングルトン）・どこからでも直接サーバ本数ぶん（並列Claudeでも共有・増えない）
Tier2 フォルダ内デフォルト全7orgそのorgフォルダで direnv 既定ゼロ（既存・在席時のみ）
Tier3 オンデマンド横断全7org＋タイプBcross-org-search（必要時に子起動）低（呼ぶ時だけ）
「起動するたびにサーバが立つ?」への答え: いいえ。常駐サーバはポート固定のHTTPシングルトンで、SessionStart hook は冪等（既に healthy なら何もせず即終了）。 並列で複数の Claude Code を開いても、全部が同じ1プロセスに接続します（OAuth refresh_token の奪い合いを避けるための設計）。常時コストは「Tier1サーバの本数」だけで、Claudeの同時起動数では増えません（例: Tier1=2本なら、何個 Claude を開いても常駐は2プロセス）。
※対照: 各org の Tier2 サーバ（stdio, .mcp.json）はセッション毎に起動しますが、フォルダに居る間だけ・終了で消えるので蓄積しません。
結論: どのorg（OpenHeart・uSonar 含む）も Tier2/3 で必ず到達できる。常駐（Tier1）は日常的にフォルダ外から触る数個だけに絞る。8org全部を常時・直接にしたい場合は MCPゲートウェイ（MetaMCP等）で1エンドポイント集約が解（org数が多い時の定石）。
要決定: どのorgを Tier1（常駐・直接）にするか。最小なら frogiraffe（=個人）Google ＋ HDT Google の2つ。OpenHeart / uSonar / Lagoon 等をフォルダ外から頻繁に触るなら Tier1 追加候補です。

HDTのSlack/Notionだけは経路が違う

HDT の Slack/Notion は Team Connectors（HDT の Claude アカウント＝軸A に紐づく）で提供されるため、 個人セッションから直接は呼べません。ここは cross-org-search が direnv exec HDT claude -p でHDTのClaudeアカウントの子プロセスを起動して読取し、結果だけ返します（Q2「読取は許可」に合致）。

なぜHDTだけ方式が違うのか — 認証情報が2タイプある

「HDTだけ別扱いなのは不自然では？」という疑問への答え。差は美学ではなく認証情報のタイプの違いです。 手元にポータブルな秘密があるタイプA（個人/HDT Google・frogiraffe Slack/Notion）は並列に置けますが、 claude.aiのHDT Teamログインに束縛されるタイプB（HDT Slack/Notion）はそのままでは並列化できず、子プロセス経由でのみ読めます。

含意: 「全部並列が綺麗」は正当な発想（業界では MCPアグリゲータ/ゲートウェイ と呼ばれ、ツールをサーバ名でnamespaceする）。 ただしHDT Slack/Notionをフラット化するには結局トークン発行が必要で、会社ポリシー次第。 データアカウントが数個の今は、名前付きHTTPサーバを並列に足す軽量パターン（あなたが既にやっている方式）が最適です。
まとめ: 「Claude が選ぶ」＝(a) タイプAのデータアカウント（個人/HDT Google・frogiraffe Slack/Notion 等）は専用サーバで直接呼ぶ＋(b) cross-org-search はタイプB(HDT Slack/Notion)と多org一括集約だけ。HDTのClaudeアカウント分離（軸A）はそのまま維持されます。

3.5 実行時の動作イメージ 既定のデータアカウント ＋ 必要に応じて越境

「データアカウントを分けて管理すると、実際どう動くのか」を具体化します。狙う挙動は1行で —

挙動の原則: 各フォルダには デフォルトのデータアカウントがある（明示しなければ黙ってそれを使う）。 そのうえで、全体の予定参照のように他のデータアカウントが必要な時だけ、Claude が判断して越境し、必要なデータアカウントだけを参照・実行します。

① ルーティングの判断フロー（毎回これが回る）

ポイントは「明示が無ければ既定のデータアカウント」「越境は依頼内容がトリガー」。 「全体の」「両方」「HDTの」のような語、あるいは“そのデータアカウントでしか持てない情報”を求められたとき、Claude は経路2・3へ踏み出します。

② 具体シナリオ（フラッグシップ: 個人とHDTのカレンダー横断で予定作成）

nanokit（個人スコープ）で「来週、個人とHDT両方の予定を見て空き枠に打合せを入れて」と頼んだ場合の実際の往復です。

注目点は2つ。(1) 個人スコープのまま HDT Google（:8324）に手が届いている＝越境が起きている。 (2) 書き込み（予定作成）の前に確認を一度挟んでいる＝越境write の事故防止。

③ 3つのケースで挙動を確認

いる場所依頼Claudeの動き越境
frogiraffe/「frogiraffeのNotionの今日のタスクは？」デフォルトの frogiraffe Notion を直接読むなし
nanokit/「来週、個人とHDTの予定で空き枠に打合せを」個人:8322＋HDT:8324 を両方読み→空き算出→確認後に両方へ作成あり(タイプA・直接)
nanokit/「frogiraffeのSlackで未読ある？」mcp__slack-frogiraffe__* を直接呼ぶ（cross-org-search 不要）あり(タイプA・直接)
nanokit/「HDTのSlackで今週のメンション拾って」cross-org-search が ~/.claude-hdt の子プロセスでHDT Slackを読取→要約あり(タイプB・子経由)
動作上の大事な3点: (1) 「常設」でも遅延ロードなので普段は文脈を圧迫しない（Claudeが必要と判断した時だけそのツール群を引く）。 (2) 明示しなければ既定のデータアカウント＝「迷ったらこれ」で安全側。 (3) 越境の書き込み（他のデータアカウントのカレンダー作成・Slack投稿など）は確認を挟む設計にでき、誤爆を防げる。
HDTは別、を維持: このシナリオで越境しているのは軸B（データ＝Google OAuth）だけ。 軸A（Claudeのログイン/課金）は個人のまま。HDT内で作業する時だけ ~/.claude-hdt のHDT課金になる分離は崩れません。

4. 狙い① 視覚確認の積極化 フック拡張 + 強命令 + 専用コマンド

Maxプランで画像認識を惜しまず使いたい、という要望。現状の plot-vision-reminder.sh は プロット画像専用（matplotlib や画像ファイル書き込みにしか発火しない）なので、対象を広げます。 選んでいただいた「フック＋強命令＋専用コマンド」の3点セットで、作る→描画→スクショ→自分で見て確認の往復を標準化します。

起点中身触るファイル
フック拡張フロントエンド編集（html/css/jsx/tsx/vue/svelte）後、および Claude-in-Chrome 操作後に「スクショ→Readで確認」を注入plot-vision-reminder.sh → vision-reminder.sh に一般化、settings.json の PostToolUse matcher 追加
強命令CLAUDE.md に「Maxなので視覚入力を惜しまない。UI/図/ブラウザ/生成画像に触れたら完了前に必ずスクショして Read で自己検証」nanokit/claude/CLAUDE.md
専用コマンド/visual-verify <url|file>：スクショ→チェックリスト採点（崩れ/はみ出し/重なり/コントラスト）→NGなら修正提案新規 nanokit/claude/skills/visual-verify/
実証: この資料の図も、まさにこのループで作りました（D2→SVG→PNG化→Readで実際に視認して可読性・コントラスト・崩れを確認）。

5. 狙い③ 自動メモリ・パーソナライゼーション 終了フックで抽出 + 全プロジェクト共通化

「好み・個人情報を能動的に集めて、どのプロジェクトの回答にも反映し、自動で回る」要望。 ここで設計上の落とし穴が1つあります。ハーネス組込みのメモリは プロジェクト別（~/.claude/projects/<proj>/memory/）なので、 nanokit で覚えた好みは他プロジェクトに出ません。「どこでも反映」にはグローバル層が要るのです。

層置き場所書き手読み込み範囲
グローバル個人知識~/.claude/memory/personal/（新）終了フック誘導でモデルが追記全プロジェクト（CLAUDE.md から @import）
確定した好みnanokit/claude/CLAUDE.md の「ユーザー像」節人手で昇格（自動編集はしない）全プロジェクト
プロジェクト固有既存 per-project MEMORY.md既存の自動追記そのプロジェクトのみ

自動で回す仕掛け

  • 終了フック（Stop） — memory-extract-reminder.sh：セッション終了時に「恒久的な好み/個人事実を抽出し、全体に効くものは ~/.claude/memory/personal/、案件固有は per-project へ保存せよ」と注入。stop_hook_active ガード＋クールダウンで無限ループ防止。
  • ネイティブ Auto memory 有効化 — settings.json に autoMemoryEnabled を設定（Claude が訂正・好みを自動メモ）。
  • 強命令 — CLAUDE.md に「ユーザーの好み・背景を能動的に収集し回答へ反映」を明記。
安全側の判断: フックは「保存せよ」と促すだけで、dotter管理下の CLAUDE.md をスクリプトが自動で書き換えることはしません。確定した好みの CLAUDE.md への昇格は人手レビューを挟みます。
